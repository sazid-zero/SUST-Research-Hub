ABSTRACT  
  
The SUST Research Hub is a comprehensive digital platform designed to transform how 
academic research — including theses, publications, projects, datasets, and machine 
learning models — is stored, discovered, and managed within Shahjalal University of 
Science and Technology (SUST). It replaces the traditional practice of archiving research 
outputs as physical hardcopies, which inherently makes them difficult to search, verify 
for originality, and build upon. Traditional, paper -based or scattered digital sub mission 
processes often result in disorganized archives, inconsistent approval workflows, and 
limited accessibility for students, supervisors, and administrative staff. These challenges 
reduce research visibility, complicate the validation of new thesis to pics, and hinder 
institutional knowledge preservation.  
This project presents the design, development, and complete implementation of a 
modernized Thesis and Research Repository Management System. Developed as a full -
scale web application, the system successfully streamlines the end -to-end lifecycle of 
thesis /publication/project  submission, storage, supervisor approval, and public 
discovery across the departments of SUST.  
The fully implemented platform features robust foundational and advanced capabilities, 
including a secure role -based access control (RBAC) system for students, supervisors, 
and administrators, accompanied by custom session management and automated email 
notifications. It encompasses complete supervisor –student workflows, enabling 
seamless thesis /project  submission pipelines, iterative review systems, and direct 
feedback mechanisms. Furthermore, the repository has been expanded beyond 
traditional documents to fully support the hosting and management of multi -author 
publications, research datasets, and mac hine learning models, complete with popularity 
tracking and analytics.  
A responsive, component -based frontend architecture ensures high -performance 
accessibility across all devices. This report documents the problem context, system 
requirements, architectural design, implemented modules, and testing phases. By 
adhering to mod ern web development best practices, the SUST Research Hub delivers a 
secure, scalable, and fully functional academic repository that actively preserves 
institutional knowledge and fosters future research collaboration.  
Keywords:  Thesis Repository, Research Management, Academic Platform, Web 
Application, Next.js,  PostgreSQL, Role -Based Access Control, Publications , Projects,  
Machine Learning Models, Datase ts.

  CHAPTER 1  
INTRODUCTION  
   
1.1 Background  
Universities produce a large volume of academic research in the form of undergraduate 
theses, projects, and faculty -led studies. These research outcomes are valuable resources, 
yet in most institutions, including SUST, they are traditionally stored as phys ical 
hardcopies in departmental libraries, making them difficult to access and underutilized. 
Students often struggle to explore previous work, supervisors cannot easily verify whether 
a proposed topic has already been completed, and opportunities to exten d or improve 
existing research are frequently missed.  
A centralized digital research repository provides a modern alternative by supporting 
long -term preservation, easy academic reference, and verification of new research topics 
through open access to structured metadata, full -text documents, and search tools . 
The SUST Research Hub is designed as a modern web -based platform that meets these 
institutional needs  and has  grown into a  research ecosystem  that covers  theses,  
publications, projects, datasets, and machine learning models . By adhering to rigorous 
engineering standards, the platform successfully establishes a robust, highly accessible, 
and fully integrated environment that transforms how university research is preserved, 
discovered, and built upon.  
 
1.2 Problem Statement  
Traditional thesis submission and storage processes face several challenges:  
• Research is stored only as physical hardcopies in libraries  
• Limited accessibility to past research  
• No simple way  or digital solution available  for students  and supervisors  to access 
previous research   
• Inefficient verification of topic originality for new research  
• No administrative tools for monitoring research activity  
• Supervisors cannot efficiently check if a topic has already been completed  
• Limited ability for new researchers to build upon existing work .  
These issues  highlight the need for a secure, centralized  and scalable  digital  platform.

SYSTEM DESIGN  
 
 4.1 Design Methodology   
The system design follows IEEE 42010 architectural principles and modern industry 
practice. A modular, layered architecture ensures scalability, maintainability, and clear 
separation of concerns, and has proven flexible enough to absorb the Publications, 
Projects, Datasets, and Models modules without restructuring the core schema.  
 
4.2 System Architecture View  
4.2.1 Logical Architecture  
• Presentation Layer: UI components (Next.js, React)  
• Application Layer: Server Actions, business logic  
• Data Access Layer: database queries, indexing  
• Persistence & Storage Layer: PostgreSQL (Neon) + Cloudinary  
4.2.2 Architectural Style  
• Client –Server architecture  
• Modular monolith (potential future evolution toward services)  
• Server Actions as the primary backend mechanism, with limited REST endpoints 
where needed

  4.5.3 Normalization  The schema maintains 1NF (atomic attributes), 2NF (UUID

4.5.4 Indexing Strategy

4.6 Security Design  
4.6.1 Authentication   
Authentication is session -based. On login, a 32 -byte cryptographically secure random 
token is generated and stored in the sessions table with a 7 -day expiry, mapped to the 
user's ID. The token is returned to the client in an HTTP -only session  token cookie. 
Every authenticated request is validated with a database join between users and 
sessions, checking existence and expiry before the request proceeds. This design is 
deliberate: it keeps sessions server -revocable and avoids the refresh -token co mplexity of 
stateless JWTs, at the cost of a database lookup per authenticated request — an 
acceptable trade -off at the current scale.  
4.6.2 Authorization (RBAC)  
• Students: access to browse, submission, and personal dashboard views  
• Supervisors: access to supervision and review features  
• Admins: full user and content management  
4.6.3 Threat Mitigation  
• SQL injection prevention via parameterized queries  
• Audit logs for accountability  
• Account approval gating prevents unverified self -registration from gaining access  
• Passwords never stored in plaintext (bcrypt)  
No formal penetration test, dependency audit, or third -party security review has been 
performed.

CHAPTER 5  
IMPLEMENTATION  
 
5.1 Overview  
This phase extended the foundational backend and frontend built in v1 with four new 
content modules and a documented, verified authentication mechanism. The goal 
remained a functional, honestly -scoped base rather than a feature -complete platform.  
Implementation stack: Next.js (App Router), React 19 + TypeScript, PostgreSQL (Neon), 
Tailwind CSS, Resend, Cloudinary, bcrypt.  
5.2 Implemented Features  
1. Role -Based User Registration & Admin Approval   
Unchanged in mechanism from v1: registration requests are validated server -side, 
stored in registration_requests, and reviewed by admins, who approve/reject with 
optional notes. Approved users are inserted into users with a bcrypt -hashed password.  
2. Session -Based Authentication  
Login issues a 32 -byte random session token, stored in the sessions table with a 7 -day 
expiry and returned via an HTTP -only session_token  cookie. Every authenticated 
request re -validates the token against the database. Unauthorized users cannot access 
protected routes.

Code Snippet 1: Session Verification Middleware  
export async function verifySession(token: string): Promise<User | null> {  
  const results = await sql`  
    SELECT u.* FROM users u  
    JOIN sessions s ON u.id = s.user_id  
    WHERE s.token = ${token} AND s.expires_at > NOW()  
    LIMIT 1  
  ` 
  return (results[0] as User) || null  
}
